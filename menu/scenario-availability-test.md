# scenario: availability test

Назначение: свежая ЛЛМ-сессия проверяет, какие файлы из протокольного репо реально доступны через её фетч-инструмент, и выдаёт оператору структурированный отчёт.

after_completion: return_to_menu

## Что ты делаешь сразу

Прочитав этот файл, скажи оператору одной строкой: "Сценарий: тест доступности файлов. Сейчас прочитаю список файлов из README и пройду по нему. Это займёт несколько ходов." Не спрашивай разрешения — просто работай.

## Шаг 1. Получи список файлов

Открой URL `https://raw.githubusercontent.com/iamweasel89/protocols/main/README.md?nocache=1`. В нём найди секцию `### Direct entry URLs`. Извлеки оттуда полный список ссылок (каждая строка вида `- https://...`).

Если по какой-то причине список не извлекается (фетч не прошёл, секция не найдена, формат не распознан) — сообщи оператору точно, что сорвалось, и предложи передать список вручную. Не выдумывай URL.

## Шаг 2. Пройди по списку

Для каждого URL из списка:

1. Попробуй его прочитать своим фетч-инструментом.
2. Если получилось — составь одну короткую фразу, описывающую содержимое файла (что это, для чего). Это и есть "краткое содержание".
3. Зафиксируй статус:
   - `read` — файл прочитан, краткое содержание составлено.
   - `failed` — фетч не прошёл (404, отказ, ошибка инструмента).
   - `unclear` — фетч прошёл, но содержимое неполное, нечитаемое или ты не можешь дать осмысленное summary.
4. Если есть сложности с фетчем (например, получено только часть файла, или формат неожиданный, или содержимое обрезано) — запомни это, добавишь в `model_self_report` отчёта.

Не торопись. Если за один ход успеваешь обработать 3–5 файлов — это нормально. Сообщай оператору о прогрессе короткими репликами вида "обработала N из M".

## Шаг 3. Собери отчёт

Когда список пройден полностью, выдай оператору один JSON-объект следующей формы:

```json
{
  "report_id": "availability-test-YYYY-MM-DD",
  "created": "<сегодняшняя дата>",
  "model": "<имя модели, как ты её знаешь>",
  "source_list_url": "<URL README, откуда взят список>",
  "items": [
    {
      "url": "<полный URL>",
      "status": "read | failed | unclear",
      "summary": "<краткое содержание или null если status != read>",
      "note": "<опционально: заметка, если что-то особенное>"
    }
  ],
  "model_self_report": [
    "<строки про трудности с фетчем, обрывы, неожиданные форматы — каждая отдельной строкой>"
  ],
  "not_canonized": [
    "<если в ходе работы возникали идеи/предположения, которые ты не вносила в отчёт, но стоит зафиксировать — пиши сюда>"
  ]
}
```

Только JSON, без прозы вокруг. Это финальный артефакт сценария.

## Шаг 4. Возврат

После того как отчёт выдан, скажи оператору одной фразой: "Отчёт готов. Возвращаюсь в меню." Затем покажи список пунктов меню снова — с пометкой `(draft, может не работать)` для draft-пунктов, как делал диспетчер при загрузке.

## Дисциплина

- Не выдумывай URL. Если списка нет — скажи об этом.
- Не выдумывай содержимое файла. Если фетч обрезан — статус `unclear`, объяснение в `model_self_report`.
- Не пропускай файлы молча. Каждый URL из списка должен попасть в отчёт.
- Не объединяй несколько файлов в одну запись. Один URL — одна запись.

## Что не входит в этот сценарий

- Анализ зависимостей между файлами.
- Рекомендации по правкам.
- Сравнение с предыдущими прогонами.

Это всё — отдельные сценарии в будущем, если понадобятся.
